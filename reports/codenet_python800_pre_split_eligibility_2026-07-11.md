# Project CodeNet Python800: pre-split eligibility audit

Дата аудита: 11 июля 2026 г.

Статус: полный D0-D3 и statement-based D4 precheck выполнены до генерации
problem split и без вычисления retrieval-метрик. Окончательный eligibility
manifest остаётся незакрытым до проверки официальной карты идентичных задач из
полного каталога `derived` Project CodeNet.

- программ: 240 000, по 300 программ на задачу;
- транспортная копия benchmark-архива:

## 2. Каскад очистки

### D0. Нормализованный исходный текст

Файл декодируется согласно Python encoding cookie, преобразуется в UTF-8,
переводы строк приводятся к `LF`, хвостовые пробелы и табуляции удаляются.
Совпадение определяется по SHA-256 полного нормализованного текста.

### D1. Точный поток лексем

Из Python token stream исключаются comments, whitespace, indent/dedent и
line-boundary tokens. Значения идентификаторов, операторов и литералов
сохраняются. Совпадение определяется по SHA-256 последовательности
`token_type:token_value`.

### D2. Alpha-нормализованный AST

Идентификаторы заменяются детерминированно по порядку первого появления,
атрибуты и импорты получают отдельные пространства имён, литералы заменяются
типизированными placeholders. Хешируется сериализация AST без координат
исходного текста.

Глобальные компоненты D0-D2 строятся до split. Один канонический представитель
выбирается по лексикографически минимальному относительному пути.

### D3. Near-duplicate token 5-grams

- представление: множество последовательных D1 token 5-grams;
- MinHash: 256 координат, схема `affine32`, seed `20260711`;
- LSH: 32 bands по 8 rows;
- роль LSH: только генерация пар-кандидатов;
- окончательное ребро: точный set-Jaccard не ниже 0,90;
- транзитивность: связные компоненты проверенного графа.

При сходстве ровно 0,90 теоретическая вероятность попадания пары хотя бы в
одну LSH-полосу равна

`1 - (1 - 0.90^8)^32`,

то есть превышает `0.99999998`. Это характеристика candidate generator, а не
замена точной проверки.

### D4. Задачи

До получения официальной карты применены два машинно проверяемых правила:

1. объединение по не менее чем пяти общим D2-программам при доле не менее 5%
   меньшей задачи;
2. одинаковый SHA-256 видимого текста HTML-условия после Unicode NFKC,
   `casefold` и свёртки whitespace.

Cross-problem D3-компонента также должна целиком попасть в один split, поэтому
связанные ею problem IDs объединяются в одну единицу назначения.

## 4. Решение по воротам

### Parse gate

На уровне языка gate `parse rate >= 95%` пройден: `98,1725%`. На уровне
отдельных задач 68 задач имеют меньший parse rate. Текущая спецификация
формулирует порог для языка и отдельно требует не менее 64 принятых программ
на cluster; это требование проходит каждый из 773 кластеров. До Stage A
необходимо письменно подтвердить именно такую единицу parse-rate gate. После
регистрации менять её запрещено.

### Precision/power gate

Для схемы `300 train / 100 validation / at least 364 test` требуется не менее
764 clusters. Текущий результат 773 оставляет запас девять кластеров. Поэтому:

- оценочный test с не менее чем 364 clusters пока возможен;
- точное номинальное разбиение `300/100/400` уже невозможно после объединения
  связанных задач;
- применяется только заранее определённое пропорциональное/greedy назначение;
- если официальный D4 уменьшит число clusters ниже 764, practically meaningful
  superiority claim для `delta*=0.010 MAP@R` не открывается без новой Stage A.

### Seal gate

Не пройден намеренно. Публичный beacon split ещё не создавался, test labels не
открывались, retrieval-метрики не вычислялись. Текущий результат не является
экспериментом качества Code2Hyp.

## 6. Следующее обязательное действие

1. Сопоставить 800 problem IDs с официальной картой identical problem
   clusters из `Project_CodeNet/derived`.
2. Добавить user/time metadata только как D5 diagnostic, не меняя основной
   D0-D4 estimand. Выполнено: полное author removal признано невалидным из-за
   средней attrition 96,02%; фиксируется user-distinct выбор внутри кластера и

3. Сформировать окончательный список eligible cluster IDs и его SHA-256.
4. Опубликовать Stage A design registration.
5. Взять первый следующий pulse NIST Randomness Beacon и только после этого
   создать HMAC split manifest.

## 7. D5 author diagnostic

Официальные metadata сопоставлены со всеми 240 000 benchmark submissions без
пропусков и дублей; все имеют статус `Accepted`. После D0-D3 индекс содержит
213 550 программ и 14 036 повторно хешированных пользователей. 9 402
пользователя встречаются более чем в одном problem cluster; на них приходится
208 720 программ.

Выполнены 1 000 planning assignments с seeds `100000..100999`. Для 773
clusters метод Гамильтона задаёт квоты `290/97/386`. Если удалить из train все
решения пользователей, встречающихся в test, получаем:

| Показатель | Среднее | Медиана | 2,5% | 97,5% |
|---|---:|---:|---:|---:|
| Доля сохранённых train-программ | 3,982% | 3,958% | 3,305% | 4,823% |
| Train clusters с не менее чем 64 программами | 4,403 | 4 | 1 | 10 |
| Минимум программ в train cluster | 0 | 0 | 0 | 0 |

Во всех 773 problem clusters имеется не менее 92 разных пользователей,
медиана равна 260. Поэтому до Stage A фиксируется user-distinct sampling
внутри кластера, а глобальный author removal остаётся только attrition
diagnostic и не порождает альтернативный model-quality результат.
